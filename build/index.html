DPG Hacktoberfest Guidance

This video contains a walk-through of the steps to get your repository ready for Hacktoberfest.

We understand that getting your project ready for the tidal wave of Hacktoberfest contributors can feel overwhelming and it can be difficult to figure out where to start. To put your concerns at ease we have put together some best practices for you to remove the guesswork in what is needed!

In the video and summary below we go through the various parts of getting your GitHub repository ready for contributors. You can think of each of the key takeaway sections as todo lists, which is why we have summarized each section into bullet points so you can easily know what you need to do.

We recommend watching the video we have put together that goes through and explains each of the required sections. Then if you have any questions you can use this page as a reference.

GitHub Repository Insights

The Insights tab of your GitHub repository gives you a comprehensive, and high-level, overview of what is happening in your repository. You can find easily at hand, recent activity, traffic, contributor information, dependency, download information and community health.

The first step in ensuring your repository is ready for contributors can be found in the community standards section. The community standards ensure that your repository has all the required items and files that new contributors to your project will be looking for. Once you see all the sections checked off you know you are ready to go!

Insights Key Takeaways

  • Get familiar with the Insights tab of your repository and the various sections it contains.
  • Check the community standards section of the insights tab and work through each of the uncompleted items until they are all checked off.
  • Here are some example pull request templates and issue templates to help you get started.
  • Creating contributing guidelines is a larger undertaking and has its own section below.

Contribution Guidelines

New contributors to your project are going to expect a CONTRIBUTING.md file in your project that explains to them the ins and outs of contributing. This file should contain, at a minimum, the process by which contributors will submit their code contributions and the requirements of doing that. You should also clearly lay out if your project expects contributors to fork the project and then submit a pull request or if your project adds contributors to the project so they can submit a branch. If your project needs specific software or requires specific operating systems to be able to develop and contribute it should be clearly laid out in this file.

Here is an excellent example of a good contributing file and here is another one.

Contributor Key Takeaways

  • You should have clear instructions on how a contributor claims or proposes something to be worked on and how they will be acknowledged as the person to do that work.
  • The process of submitting a pull request to your project should be clearly laid out.
  • Software and operating system requiremnts to contribute, if any, should also be listed. Let folks know if they need docker, linux, or something else.

Issues

Issues are where contributors come and see what tasks there are and how they can contribute to your project. This is also an opportunity to build a relationship with contributors by connecting the work they will be doing to the mission or stakeholders your project is serving. A well written issue should contain three key sections. The first is that it should explain the task that is to be completed by the contributor that is ideally written in a way that ties the work to be done back to social sector work of the project. Second is to add in a section with the information that will help the contributor, who may not have a full understanding of the domain, successful solve the problem. Finally, it should be clearly spelled out what is required to successful complete the issue. Here is an example of an issue containing a summary, additional information, and completion details.

If your project allows contributors who are not part of your core team to submit issues we recomment that your issue templates adds in additional sections. We recommend having a section for the contributor to submit a justification for the requested issue to be added as well as a section for the contributor to add in any research or code details that may be relevant in solving this issue. Here is an example of a contributor submitted issue containing a justification and code details.

Issues Takeaways

  • Issues should have a clear and understandable descripion to someone who is unfamiliar with the project of the work that is to be completed. Bonus points if you add in pictures.
  • Any technical details or specific software requirements needed to complete the issue for the contributor should be explicitly stated.
  • There should be no ambiguity in what needs to be submitted by the contributor for the issue to be considered completed.

Labels

Labels provide an easy way for contributors to identify both the types issues that they are interested in working on as well as if those issues are ready to be worked on. We recommend that you have a set of labels that you can attach to issues that provide a high level overview of the type of work that needs to be done. For example, if you are working on a project that is a website you may have labels for "Frontend", "Backend", "Design", "Documentation", etc. With labels like this contributors can easily identify the type of work they are interested in combine that with the labels like "Ready to Work" and "Help Wanted" that indicate if the issue is ready to be worked on and your project will be a breeze for contributors to navigate.

By creating a custom "Hacktoberfest" label you can also make it easy for contributors to identify issues that are specifically designated for Hacktoberfest. During October contributors look specifically for this label as contributing to projects that are participating in Hacktoberfest is rewarded with some developer swag.

If you use GitHub project boards to manage your issues you can take advantage of GitHub actions to automatically add labels to issues as they are added to the project board. You can also use GitHub actions to automatically add labels to issues that have been inactive for a certain amount of time as well as adding or removing Help Wanted labels when an issue becomes available or claimed.

Labels Key Takeaways

  • Any issue that you'd like a contributor to work on should have a "Help Wanted" label. This label adds to a count of open issues in project lists.
  • If you are participating in Hacktoberfest you should have a "Hacktoberfest" label that you can add to issues that you'd like to be worked on during October.
  • There are a number of GitHub actions that can be used to automatically add/remove labels to issues based on the project board they are in or if they have been inactive for a certain amount of time. A good starting place is this action, this action and this action.

Hacktoberfest Etiquette

Hacktoberfest is an amazing, and hectic, time. Some projects see a huge influx of contributors and it can be hard to keep up with the volume of issues and pull requests. It is recommended that you have a set of guidelines for contributors to follow to make sure that they are contributing in a way that is helpful to your project and that they know you will be responsive to your contributors.

We recommend that you have a section in your README that outlines any special guidelines for Hacktoberfest contributors. This section should also include how soon they can expect to have their questions answered and their pull requests reviewed.

Etiquette Key Takeaways

  • Hacktoberfest is hectic for contributors as well as project maintainers. Set clear expectations in your README around Hacktoberfest.
  • If your project holds "office hours" or "drop in hours" you should publish them for contributors. It is a great way to build community and help build community around your project.
  • Don't forget to sign up for Hacktoberfest as a maintainer!
